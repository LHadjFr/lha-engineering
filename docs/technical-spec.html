LHA Engineering — Spécification Technique

Version 1.1 Avril 2026 Statut : Production-ready Auteur : LHA Engineering / GitHub Copilot

1. Introduction

Ce document décrit l'architecture, les choix technologiques et les spécifications fonctionnelles du site web de LHA Engineering. Il est destiné à tout développeur reprenant le projet, intervenant sur l'infra ou souhaitant comprendre les décisions prises.

Le site est une application React full-stack Dockerisée qui remplit trois rôles : vitrine commerciale, présentation des labels de l'entreprise avec pages dédiées, et formulaire de contact connecté à un backend Express avec envoi SMTP réel.

2. Objectifs

  • Présenter l'expertise de LHA Engineering autour de trois labels métier.
  • Proposer une page dédiée par label avec contenu détaillé.
  • Fournir un formulaire de contact fonctionnel, protégé, avec envoi SMTP ou fallback disque.
  • Isoler intégralement l'environnement de développement dans Docker — aucune installation Node locale requise.
  • Déploiement production sur serveur dédié : Nginx + TLS + Let's Encrypt.

6. Backend API

Fichier unique : server/index.mjs. Framework : Express 4. Runtime : Node 22 ESM.

Endpoints

Schéma de la requête POST /api/contact

{
  "name":          string  // requis, non vide
  "email":         string  // requis, format email valide
  "company":       string  // optionnel
  "subject":       string  // requis, non vide
  "message":       string  // requis, non vide
  "website":       string  // champ honeypot — doit être vide
  "formStartedAt": number  // timestamp ms côté client — doit être ≥ 4 000 ms avant la soumission
}

Comportement d'envoi

  • Si SMTP_HOST est défini et différent de smtp.example.com, et que SMTP_FROM et CONTACT_TO sont renseignés → envoi SMTP via Nodemailer → réponse {"ok":true,"mode":"email"}.
  • Sinon → stockage JSON dans server/submissions/<timestamp>-<slug-nom>.json → réponse {"ok":true,"mode":"stored"}.

Format du fichier de soumission stocké

{
  "name": "Prénom Nom",
  "email": "expediteur@exemple.fr",
  "company": "Nom société",
  "subject": "Sujet du message",
  "message": "Corps du message",
  "receivedAt": "2026-04-20T14:18:50.829Z",
  "requesterIp": "127.0.0.1"
}

Réseau production

Un réseau bridge Docker nommé lha-engineering-net est créé automatiquement par scripts/prod-up.sh. Les deux conteneurs prod y sont attachés. Nginx résout lha-engineering-api par son nom de conteneur via l'upstream Nginx.

Note : En développement, le code source est monté en volume (--volume $(PWD):/app). Les node_modules sont dans un volume nommé séparé (lha_node_modules) pour éviter les conflits avec l'hôte Linux.

8. Stack Frontend — Justification des choix

React 19 + TypeScript

Framework UI déclaratif. TypeScript pour la sûreté des types à l'échelle du projet. React 19 est la dernière version stable, compatible avec Vite 8 et React Router 7.

Vite 8

Bundler de développement ultra-rapide avec HMR natif. En production, génère un bundle optimisé avec tree-shaking. Configuré avec usePolling: true pour la compatibilité inotify/Docker volumes.

Tailwind CSS v4

Utilisation via le plugin Vite officiel (@tailwindcss/vite), sans tailwind.config.js. Configuration par import CSS direct dans src/index.css. Choix justifié par la vitesse de prototypage d'un UI complexe sans CSS custom.

React Router v7

Routing côté client pour les pages dédiées par label (/labels/:labelId). Gestion du scroll restauré via ScrollManager (hook useLocation + useEffect).

Lucide React 0.468

Bibliothèque d'icônes SVG légère, tree-shakable. Utilisée pour les icônes de navigation, de service et du formulaire.

9. Stack Backend — Justification des choix

Node.js 22 ESM (.mjs)

Runtime JavaScript serveur. Fichier unique server/index.mjs — pas de build nécessaire. ESM natif pour cohérence avec le reste du projet ("type": "module" dans package.json).

Express 4

Framework HTTP minimaliste. Justifié par la simplicité du besoin (2 routes). app.set('trust proxy', true) pour lire correctement l'IP réelle derrière Nginx.

Nodemailer 6

Envoi SMTP standard. Aucune dépendance propriétaire. Compatible avec tout fournisseur SMTP (Gandi, OVH, Brevo, SendGrid…). L'objet transporter est créé à chaque envoi pour éviter les connexions persistantes en contexte serverless-like.

Concurrently

Lance Vite et l'API Express en parallèle dans le même conteneur dev avec npm run dev → concurrently -k "npm:dev:api" "npm:dev:web".

10. Conteneurisation — Justification des choix

Ubuntu 24.04 pour le dev (Dockerfile.dev)

Choix explicite du client : l'environnement de dev doit ressembler à un Ubuntu complet pour faciliter les interventions manuelles (make shell). Node 22 est installé via NodeSource pour contrôler précisément la version.

node:22-bookworm-slim pour la prod API

Image Debian Slim officielle : légère, sécurisée, compatible production. Seules les dépendances runtime (npm install --omit=dev) sont installées.

Build multi-stage pour l'image web

Stage 1 (node:22-bookworm-slim) : build React → dist/. Stage 2 (nginx:1.27-alpine) : copie du dist uniquement, aucune dépendance Node dans l'image finale. Résultat : image finale légère (~40 Mo).

Docker pur (pas de Compose)

Contrainte explicite du client. Toute l'orchestration est gérée via des scripts shell (scripts/prod-up.sh, prod-down.sh) et un Makefile.

11. Pages et navigation

La navigation principale est assurée par SiteHeader via des liens hash (/#expertise, /#labels, /#contact) qui scroll automatiquement grâce au composant ScrollManager.

Chaque label dispose d'une page dédiée accessible depuis le bouton "Détails" de chaque carte. La donnée est centralisée dans src/site-data.ts et partagée entre la carte d'accueil et la page dédiée.

13. Protection anti-spam

Trois couches complémentaires, sans CAPTCHA ni dépendance externe :

MécanismeOùFonctionnement
Honeypot Frontend + API Champ website masqué CSS + tabIndex="-1". Si rempli (bots) → rejet 400.
Délai minimal Frontend + API Timestamp formStartedAt enregistré au montage du composant. Si la soumission arrive moins de CONTACT_MIN_SUBMIT_TIME_MS (défaut 4 000 ms) après le chargement → rejet 400.
Rate limiting IP API Map en mémoire. Maximum CONTACT_RATE_LIMIT_MAX (défaut 5) requêtes par IP dans une fenêtre de CONTACT_RATE_LIMIT_WINDOW_MS (défaut 10 min). Au-delà → rejet 429.
Limitation connue : Le rate limiting est en mémoire processus. Il est réinitialisé au redémarrage du conteneur. Pour une protection plus robuste sur plusieurs instances, remplacer la Map par Redis.

17. Configuration SMTP (Gandi)

Valeurs à renseigner dans .env pour le provider Gandi :

SMTP_HOST=mail.gandi.net
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=contact@votre-domaine.tld
SMTP_PASSWORD=mot_de_passe_boite_mail
SMTP_FROM=LHA Engineering <contact@votre-domaine.tld>
CONTACT_TO=contact@votre-domaine.tld

Alternative SMTPS (TLS immédiat) :

SMTP_PORT=465
SMTP_SECURE=true

Décision de la logique d'envoi

La fonction isMailTransportConfigured() dans server/index.mjs active l'envoi SMTP uniquement si :

  • SMTP_HOST est défini et différent de smtp.example.com
  • SMTP_FROM est renseigné
  • CONTACT_TO est renseigné

Sinon, fallback automatique sur le stockage JSON dans server/submissions/.

18. HTTPS & Let's Encrypt

Le script scripts/prod-init-letsencrypt.sh utilise le mode standalone de Certbot : il démarre un serveur HTTP temporaire sur le port 80 pour valider le domaine. Les certificats sont placés dans deploy/letsencrypt/ et montés en lecture seule dans le conteneur Nginx.

19. Points bloquants rencontrés

Cette section documente les incidents et éléments bloquants identifiés lors du développement et du déploiement, afin d'orienter rapidement tout développeur qui reprendrait le projet.

19.1 SMTP sortant bloqué sur Railway (2026-04-24)

Statut : non résolu sur Railway — comportement inhérent à la plateforme, indépendant de la configuration SMTP.

Symptôme

Le formulaire de contact (POST /api/contact) est fonctionnel côté validation mais retourne systématiquement HTTP 500 lors de l'envoi de l'email, malgré une configuration SMTP apparemment correcte. Durée de la requête observée dans les logs Railway : ~10 070 ms, cohérente avec un timeout de connexion SMTP.

Diagnostic

  • Tentatives avec SMTP_PORT=587 (STARTTLS, SMTP_SECURE=false) et SMTP_PORT=465 (TLS, SMTP_SECURE=true) → même comportement.
  • Fournisseur SMTP : mail.gandi.net. Les identifiants et la boîte mail sont valides (testés hors Railway).
  • Hypothèse confirmée : le runtime Railway filtre/bloque les connexions TCP sortantes vers les ports SMTP (587, 465), quel que soit l'hôte cible.

Contournement appliqué

Commit 9c731eb : ajout des options connectionTimeout: 10000 et socketTimeout: 10000 dans la configuration Nodemailer de server/index.mjs pour borner le hang et obtenir une erreur propre plutôt qu'un timeout applicatif indéfini. Ce changement n'est pas une solution — il améliore uniquement la résilience.

Solutions recommandées

OptionImpact codeComplexité
Migration hébergement vers Render Aucun — mêmes variables d'env, même commandes build/start Faible
Remplacer Nodemailer SMTP par une API HTTP email (Resend, Brevo, Mailgun, SendGrid) Modification de server/index.mjs uniquement — logique d'envoi isolée Moyenne

Paramètres minimaux pour migration Render : Build npm ci && npm run build, Start npm start, Healthcheck /api/health, copie des variables d'env (sauf API_PORT).

19.2 Exposition du mot de passe SMTP

Action requise : le mot de passe SMTP Gandi a été exposé en clair lors du diagnostic (session de chat). Il doit être rotaté immédiatement dans l'interface Gandi et mis à jour dans les secrets de la plateforme d'hébergement.

Procédure de rotation :

  1. Se connecter à admin.gandi.net → Messagerie → Boîte concernée → Modifier le mot de passe.
  2. Mettre à jour la variable SMTP_PASSWORD dans les secrets Railway (ou Render) — ne jamais la committer dans .env versionné.
  3. Vérifier que .env est bien listé dans .gitignore (c'est le cas).

19.3 Canonicalisation du domaine

Le domaine canonique de production est https://lha-engineering.fr (apex, sans www). La configuration Nginx redirige https://www.lha-engineering.fr → https://lha-engineering.fr via un return 301. Le certificat Certbot couvre les deux variantes (lha-engineering.fr et www.lha-engineering.fr) grâce à la SAN du script scripts/prod-init-letsencrypt.sh.

Cohérence SEO : la balise <link rel="canonical"> dans index.html doit pointer vers l'apex (https://lha-engineering.fr), sans www, pour éviter le contenu dupliqué.
